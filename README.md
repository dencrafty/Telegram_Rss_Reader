Description:

RSS читалка для телеграм каналов.<br>
В основе парсинг страниц https://t.me через jsoup. <br>
Отсутствует авторизация, трэкеры аналитики, java скрипты. Также, по описанию библиотеки jsoup, "замыливается" фингерпринт пользователя. В связке с vpn вроде неплохой результат. <br>
Добавил проверку на скам (пока что постов вида https://t.me/+, т.е. те каналы, чей id начинается на +). Такие посты отображаются серым текстом, но с красным уведомлением.
Подгружаются в кэш recycler view. Поэтому можно взять (например 1000 за счет легковесности корутин) постов и читать где-то без интернет доступа. <br>
Отсутствуют видео, аудио, картинки - такая задумка. Формат приложения RSS подразумевает только текстовую информацию + это сэмпл тестового задания к моему cv. (при желании можно и url-ы картинок парсить).
<br>
<br>
<br>
Business logic:

Все данные из https://t.me (в перспективе можно с tgstat.com, но там ввели кучу ограничений для free). <br>
Вводите имя канала и приложение начинает с запроса единственной страницы get https://t.me/s/{имя_канала} . Таким образом получаю количество постов на канале с начала его основания. <br>
Далее все посты, начиная со свежего, вытягиваю через отдельный запрос для каждого поста, в порядке убывания номера поста get https://t.me/{имя_канала}/{номер_поста} .
<br>
<br>
<br>
Architecture:

Принципы Clean Architecture из best practices developer.android.com <br>
Только 2 слоя : Ui и Data. Domain тут излишний. <br>
<br>
  &emsp;UI layer:
  
  &emsp;Host activity с фрагментами (single-activity architecture). <br>
  &emsp;Пакет ui.paging (адаптер, пейджер и viewholder для отображения списка). <br>
  &emsp;Использую Model-View-ViewModel.


  &emsp;Data layer:

  &emsp;PagingSource данные для работы с пагинацией. <br>
  &emsp;Repository, реальзующий интерфейсы бизнес логики. <br>
  &emsp;DataSource - вытягивает данные через корутины из Jsoup api. <br>
<br>
<br>
Libraries:

JSoup parser. В библиотеке встроен http client, поэтому Retrofit не использовался. <br>
Асинхронность через корутины. <br>
Для di использовал hilt. <br>
Paging3 пагинация. Библиотека хорошо разграничивает Paging api в 2 слоях: ui и data. <br>
Для навигации между фрагментами Navigation API с рекомендуемым принципом single-activity architecture. <br>
